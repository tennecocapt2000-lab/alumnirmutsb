## เป้าหมาย
ทำให้หน้า `/admin/payment-settings` บันทึกได้เร็ว ไม่ค้าง และมีผลลัพธ์ชัดเจนหลังบันทึกสำเร็จหรือผิดพลาด

## สิ่งที่จะทำ
1. ตรวจ flow การบันทึกในหน้าแอดมินและตัดจุดที่อาจรอค้างโดยไม่จำเป็น
   - แยกกรณี “สร้างข้อมูลครั้งแรก” ออกจาก “แก้ไขข้อมูลเดิม”
   - ไม่ยิงคำสั่งอัปเดตทั้งตารางก่อนทุกครั้งถ้าไม่จำเป็น

2. ปรับ logic ให้บันทึกเป็นลำดับที่ปลอดภัย
   - บันทึก/อัปเดตแถวหลักก่อน
   - ค่อยจัดการสถานะ `is_active` ให้เหลือเพียงรายการเดียวแบบชัดเจน
   - ตรวจ error ของทุกคำสั่งฐานข้อมูลทีละขั้น แทนการปล่อยให้ pending เงียบ ๆ

3. ปรับ UX ระหว่างบันทึก
   - ปุ่มบันทึกต้องถูก disable ระหว่างทำงาน
   - แสดงข้อความสถานะที่ชัดขึ้นเมื่อกำลังบันทึก
   - ถ้าสำเร็จให้มีผลลัพธ์ต่อทันที เช่น refresh ข้อมูลฟอร์มหรือ sync ค่า id/สถานะกลับเข้าฟอร์ม
   - ถ้าล้มเหลวให้แสดงข้อความ error ที่ตรงสาเหตุจริงมากขึ้น

4. ตรวจความถูกต้องของสิทธิ์และ query ที่เกี่ยวข้อง
   - ยืนยันว่าแอดมินเขียนข้อมูลตาราง `payment_settings` ได้ครบทั้งเพิ่มและแก้ไข
   - ถ้าพบว่าติดเงื่อนไขข้อมูลเดี่ยว/active record จะปรับให้ query รองรับกรณีไม่มีข้อมูลเริ่มต้นและกรณีมีข้อมูลเดิม

5. ทดสอบผลลัพธ์หลังแก้
   - ทดสอบบันทึกครั้งแรก
   - ทดสอบแก้ไขข้อมูลเดิม
   - ทดสอบเปิด/ปิดการแสดง QR และเปลี่ยนยอดเงิน
   - ยืนยันว่าหน้า Step 4 ของผู้สมัครยังอ่านค่าปัจจุบันได้ตามเดิม

## รายละเอียดทางเทคนิค
- จุดที่น่าสงสัยที่สุดตอนนี้คือ flow บันทึกของ `src/routes/admin/payment-settings.tsx`
- ตอนนี้ตาราง `payment_settings` ยังไม่มีข้อมูล จึงต้องรองรับ first-save ให้ชัดเจน
- ตารางมี unique index สำหรับ `is_active = true` ดังนั้นการสลับ active ควรทำแบบควบคุมลำดับคำสั่ง ไม่อัปเดตกว้าง ๆ แบบเสี่ยงค้าง/ชน constraint
- จะคงการดึงข้อมูลจากฐานข้อมูลจริงต่อไป ไม่ hardcode และไม่ใช้ localStorage

## ผลลัพธ์ที่คาดหวัง
- กดบันทึกแล้วไม่ค้าง
- สำเร็จหรือผิดพลาดภายในเวลาสั้นและมี feedback ชัดเจน
- ข้อมูลการชำระเงินอัปเดตได้จริงจากหลังบ้าน และหน้า Step 4 ใช้ค่าปัจจุบันต่อได้ตาม requirement
